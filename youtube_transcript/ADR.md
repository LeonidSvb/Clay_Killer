# Architecture Decision Log — YouTube Transcript Tool

Паттерны и решения этого проекта. Используй как контекст при старте похожих Streamlit + LLM инструментов.

---

## ADR-0001 — Один файл вместо модульной структуры

**Date**: 2026-03-31
**Status**: Accepted

### Context
Личная утилита, один разработчик, нет команды. Вопрос: разбивать ли на модули (llm.py, history.py, ui.py).

### Decision
Всё в одном `app.py`. Разбивка только если файл перевалит за 600 строк и появятся реально независимые домены.

### Consequences
- **Pros**: открыл — всё видно, нет прыжков между файлами, проще давать как контекст LLM
- **Cons**: со временем растёт — нужно следить за чистотой функций

## ADR-0002 — Local JSON вместо базы данных

**Date**: 2026-03-31
**Status**: Accepted

### Context
Нужно хранить историю видео, чат-сообщения, задачи, настройки. Варианты: SQLite, Supabase, JSON-файлы.

### Decision
`history.json` для данных видео/чатов/задач, `settings.json` для конфига. Никакой БД.

### Consequences
- **Pros**: нет зависимостей, нет миграций, файл можно открыть и прочитать, легко бэкапить
- **Cons**: не масштабируется на много пользователей, нет индексов — но для личного инструмента не нужно

## ADR-0003 — Структура записи в history.json

**Date**: 2026-03-31
**Status**: Accepted

### Context
Нужно хранить всё связанное с видео в одном месте, чтобы не джойнить несколько файлов.

### Decision
Каждая запись содержит все данные видео плоско: `video_id`, `title`, `channel`, `language`, `date`, `text` (транскрипт с таймингами), `segments` (массив `{start, timestamp, text}`), `messages` (история чата), `tasks` (список задач). Один объект = одно видео = всё о нём.

### Consequences
- **Pros**: одна загрузка — всё есть, нет связей, легко читать и отлаживать
- **Cons**: файл растёт — при сотнях видео стоит рассмотреть разбивку на файлы по video_id

## ADR-0004 — Транскрипт хранится с таймингами [MM:SS]

**Date**: 2026-03-31
**Status**: Accepted

### Context
YouTube Transcript API возвращает сегменты с `start` (секунды). TextFormatter выбрасывает тайминги.

### Decision
Не использовать TextFormatter. Собирать текст вручную в формате `[MM:SS] текст`. Хранить и в `text` (строка для LLM), и в `segments` (массив для ссылок).

### Consequences
- **Pros**: LLM видит тайминги и может на них ссылаться; можно генерировать ссылки `?t=секунды`
- **Cons**: текст чуть длиннее — незначительно для современных контекстных окон

## ADR-0005 — Кликабельные таймстемпы в ответах LLM

**Date**: 2026-03-31
**Status**: Accepted

### Context
LLM в ответах пишет тайминги в разных форматах: `[02:15]`, `(2:15)`, `[1:02:35]`. Нужно делать их кликабельными.

### Decision
Post-process каждый ответ LLM через `render_timestamps(text, video_id)`: regex ловит все форматы (`[M:SS]`, `[MM:SS]`, `[HH:MM:SS]`, те же в круглых скобках), конвертирует в `[MM:SS](youtube.com/watch?v=ID&t=seconds)`.

### Consequences
- **Pros**: один клик — YouTube открывается на нужной секунде; работает для любого формата который выдаёт LLM
- **Cons**: нет

## ADR-0006 — System prompt: правило таймингов как отдельная константа

**Date**: 2026-03-31
**Status**: Accepted

### Context
Нужно чтобы LLM всегда указывал тайминги. Соблазн: прописать в каждом пресете отдельно.

### Decision
Правило таймингов вынесено в `TIMESTAMP_RULE` — отдельная строка-константа. Добавляется в конец каждого system prompt через `build_system_prompt()`. Пресеты содержат только свою логику.

### Consequences
- **Pros**: правило применяется ко всем запросам включая свободный чат; меняешь в одном месте
- **Cons**: нет

## ADR-0007 — OpenRouter с reasoning-параметром

**Date**: 2026-03-31
**Status**: Accepted

### Context
Нужна LLM для Q&A по транскриптам. Варианты: OpenAI напрямую, Anthropic, OpenRouter.

### Decision
OpenRouter как единая точка доступа. Модель по умолчанию: `openai/gpt-oss-120b` (MoE, быстрая, дешёвая). Параметр `reasoning: {effort: low/medium/high/none}` — настраивается в UI. `none` отключает reasoning для простых запросов.

### Consequences
- **Pros**: одна интеграция — любая модель; reasoning регулируется под задачу; очень дёшево (~$0.04/M input)
- **Cons**: зависимость от OpenRouter как прослойки

## ADR-0008 — Настройки в settings.json, редактируемые в UI

**Date**: 2026-03-31
**Status**: Accepted

### Context
Модель, reasoning, temperature, user context, пресеты — всё это нужно менять без правки кода.

### Decision
Все конфиги в `settings.json`. Таб "Настройки" в UI позволяет редактировать всё прямо в приложении. Пресеты хранятся как массив `{label, prompt}` — редактируемы, добавляемы, удаляемы.

### Consequences
- **Pros**: нет правки кода для смены модели или промпта; settings.json легко бэкапить
- **Cons**: нет валидации — если вписать несуществующую модель, ошибка придёт только при запросе

## ADR-0009 — Чат per-video с персистентной историей

**Date**: 2026-03-31
**Status**: Accepted

### Context
Нужен чат с контекстом каждого видео. Варианты: stateless (каждый запрос с нуля), stateful (история сохраняется).

### Decision
`messages` массив хранится в `history.json` при каждом видео. При запросе к LLM: `[system+transcript, ...messages, новый вопрос]`. `save_messages()` пишет после каждого ответа. Кнопка "Очистить чат" обнуляет массив.

### Consequences
- **Pros**: можно вернуться через день и продолжить разговор; контекст не теряется
- **Cons**: длинные сессии увеличивают стоимость запроса — при необходимости можно обрезать старые сообщения

## ADR-0010 — Пресеты как preset-driven shortcuts, не агенты

**Date**: 2026-03-31
**Status**: Accepted

### Context
Хотелось сделать "AI-агента". Вопрос: настоящий агент с тулзами и orchestration или упрощённый вариант.

### Decision
Пресеты = кнопки с захардкоженными промптами. Нажал — промпт отправляется как user message в обычный чат. Никаких тулзов, никакого multi-step reasoning, никакого orchestration.

### Consequences
- **Pros**: простота, предсказуемость, нет latency от planning; UX ощущается как агент
- **Cons**: нет автономных действий — но для Q&A по видео это не нужно

## ADR-0011 — Задачи per-video с тремя статусами

**Date**: 2026-03-31
**Status**: Accepted

### Context
Нужен способ фиксировать action items из видео и возвращаться к ним.

### Decision
`tasks` массив в каждой записи history. Структура задачи: `{id, text, status, ts_ref, created}`. Три статуса: `todo`, `done`, `dropped`. Никаких "in progress", "blocked" и т.д. — для личного инструмента избыточно. `ts_ref` — опциональный тайминг источника, конвертируется в YouTube ссылку.

### Consequences
- **Pros**: простая модель — не нужно думать куда положить задачу; экспорт в MD одной функцией
- **Cons**: нет приоритетов, дедлайнов — добавить если понадобится

## ADR-0012 — Layout: табы верхнего уровня + двухколоночный вид

**Date**: 2026-03-31
**Status**: Accepted

### Context
Приложение выросло: видео + чат + задачи + глобальный список задач + настройки. Нужна навигация.

### Decision
Три таба: "Видео" (основной workflow), "Все задачи" (глобальный список), "Настройки". Внутри "Видео": две колонки 50/50 — левая (превью, транскрипт, задачи), правая (чат с пресетами).

### Consequences
- **Pros**: каждый таб = одна задача; двухколоночник позволяет работать с задачами и чатом параллельно
- **Cons**: на узких экранах колонки сжимаются — не критично для локального инструмента
